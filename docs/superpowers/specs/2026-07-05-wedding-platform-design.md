# Дизайн: платформа свадебных приглашений (MVP-основа)

Дата: 2026-07-05. Статус: реализовано в этой же сессии по прямому указанию пользователя
(«sdelay osnovu… nuka poprobuyem») — интерактивный этап вопросов пропущен, решения зафиксированы здесь.

## Цель

Механизм от заявки до готовой страницы приглашения, без ошибок. Красивые шаблоны — позже:
пока страница приглашения просто аккуратно показывает данные из Telegram.

## Поток

1. `/start` в боте → inline-кнопка открывает Web App (`/app/`).
2. Форма: имя жениха/невесты, дата, время, локация (клик по карте или поиск), музыка
   (готовый трек или своя ссылка), шаблон (с ценой и демо «Смотреть пример»), премиум —
   именные приглашения до 20 гостей (+фикс. цена).
3. POST `/api/applications` c `initData` → сервер проверяет подпись Telegram (HMAC),
   валидирует поля, считает цену сам (клиенту не верит), пишет заявку в SQLite,
   шлёт админу сообщение с кнопками «✅ Оплачено» / «❌ Отклонить».
4. «Оплачено» (только ADMIN_CHAT_ID): генерируется уникальный slug `ali-and-zebo`
   (транслитерация ru/uz), у премиума — slug-и гостей; статус `paid`; пара получает ссылки.
5. `GET /:slug` — страница приглашения; `GET /:slug/:guest` — с обращением «Hurmatli {имя}!».
   Неизвестный guest-slug показывает общую открытку (битых ссылок не бывает).

## Стек и обоснование

- **Node 24 + Express 4 + grammY**, БД — **встроенный `node:sqlite`** (нет нативной сборки,
  нет внешнего сервера БД — минимум точек отказа).
- Web App — **чистый HTML/CSS/JS без сборщика**; карта — **Leaflet + OSM** (без API-ключей),
  геокодинг — Nominatim (best-effort: при недоступности адрес вводится руками).
- `.env` через `process.loadEnvFile` (нативно), long polling (не нужен вебхук).

## Данные

`applications`: tg_user_id, tg_username, groom_name, bride_name, wedding_date (YYYY-MM-DD),
wedding_time (HH:MM), address, lat, lng, music_type ('preset'|'custom'|'none'), music_value,
template_id, template_price, premium (0|1), premium_price, guest_names (JSON), total_price,
status ('new'|'paid'|'cancelled'), slug (UNIQUE), created_at, paid_at.
`guests`: application_id, name, slug (UNIQUE в рамках заявки).

Ссылки на карты не хранятся — детерминированно выводятся из lat/lng
(Google: `maps/search/?api=1&query=lat,lng`; Yandex: `?pt=lng,lat`).

## Надёжность

- Подпись initData обязательна (обход только с `DEV_NO_AUTH=1` для локальной разработки).
- «Оплачено» идемпотентно: UPDATE с условием `status='new'` — двойной клик безопасен.
- Callback-кнопки принимаются только от ADMIN_CHAT_ID.
- Весь пользовательский ввод экранируется в HTML (страницы и сообщения бота).
- Ошибка доставки уведомления не теряет заявку (она уже в базе).
- Зарезервированные slug-и (`app`, `api`, `demo`, …) исключены из генерации.

## Отложено сознательно

Настоящие шаблоны (точка подключения — `src/render.js` по `template_id`), реальные треки,
веб-панель админа, онлайн-оплата (сейчас оплата подтверждается админом вручную — по спеке).
